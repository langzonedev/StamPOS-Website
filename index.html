Built for actual trading

Small-business POS without unnecessary complexity.

The application is designed around the work happening at the counter: add products, take payment, print receipts, reconcile the day and move to the next customer.

Fast checkout

Touch-friendly product buttons and a focused sale flow for busy service periods.

Local sales data

SQLite storage on each POS terminal keeps core trading independent of a central server.

Receipt-ready

Simulator, Windows printer and direct network printer paths support practical setup and testing.

Admin controls

Manage products, prices, visibility, reporting periods, hardware and completed sale records.

Operational confidence

Prepared for market stalls, pop-ups and event days.

StamPOS separates normal operator tasks from technician tools. Admin handles products, sales, payments and reporting. Technician tools support storage status, database health, diagnostics and pre-event readiness checks.

Admin

Create the administrator PIN, manage products and prices, review completed sales, configure payments and select receipt printing mode.

Technician

Run storage checks, database health checks, external archive support and pre-open readiness checks before trading begins.

Event flow

Zeroise/start an event reporting period, close with an itemised Z report and retain data for vendor records.

Payments without guesswork

Linkly-first, provider-neutral EFTPOS direction.

StamPOS is shaped around a provider-neutral terminal workflow, with Linkly as the preferred first production path. The goal is clear operator feedback, persisted attempts and safe recovery when a terminal result is uncertain.

Cash-up and reconciliation

Designed for the end of the day, not just the sale.

Cash tendering uses large AUD note buttons and high-visibility change due screens. Event reporting, Z reports, receipt reprints and future card reconciliation are treated as first-class POS workflows.
